# Phase 7: Version Foundation & State Persistence - Context

**Gathered:** 2026-04-10
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase establishes the **foundation** for self-update. Zero systemd, zero GitHub polling, zero UI beyond a static version display. The deliverables unblock Phases 8–10:

1. The running app knows its own commit SHA and build timestamp (baked at build time).
2. A health-aware `/api/version` endpoint other components can probe (and the updater will probe after restart).
3. Persistent update state (`.update-state/state.json`) with atomic write semantics.
4. A Drizzle `update_runs` table for per-run audit logs.
5. A tiny UI addition: short SHA visible on the existing Settings page, hover/click for full SHA + copy.

**Explicitly NOT in Phase 7:**
- GitHub API client, ETag handling, 6h poller → Phase 8
- systemd unit, shell script, rollback pipeline → Phase 9
- Install button, live log SSE, reconnect overlay → Phase 10
- Changelog preview, release channels, update banners → never (v1.3+)

</domain>

<decisions>
## Locked Implementation Decisions

All decisions below were locked during milestone-level research and the new-milestone Q&A.

### Version baking strategy
- **Generated file, not env vars:** `src/lib/version.ts` is generated by a prebuild script and git-ignored. Single source of truth for server AND client (no `NEXT_PUBLIC_*` env indirection, no duplicated values).
- **Prebuild script:** `scripts/build/generate-version.mjs` uses `execSync('git rev-parse HEAD')` and `new Date().toISOString()` to write literal TypeScript exports.
- **Package scripts:** Add `"gen:version": "node scripts/build/generate-version.mjs"`. Chain it into `dev` and `build`:
  - `"dev": "pnpm gen:version && next dev"`
  - `"build": "pnpm gen:version && next build"`
- **Git-ignore:** Add `src/lib/version.ts` to `.gitignore` so every environment bakes its own.
- **Fallback:** If `git rev-parse` fails (not a git repo), write `"unknown"` SHA + current timestamp, don't crash the build.

### Exported shape
```ts
// src/lib/version.ts (generated)
export const CURRENT_SHA = "abc1234567890...";      // full SHA
export const CURRENT_SHA_SHORT = "abc1234";          // 7-char short
export const BUILD_TIME = "2026-04-10T12:34:56.789Z";
```

### /api/version endpoint shape
```ts
GET /api/version → 200 {
  sha: string,          // full
  shaShort: string,     // 7-char
  buildTime: string,    // ISO
  rollbackSha: string | null,   // from state.json; null on fresh install
  dbHealthy: boolean    // live SQLite probe (SELECT 1)
}
```
- **Placement:** `src/app/api/version/route.ts`, `export const runtime = "nodejs"`.
- **Health probe:** Synchronous `SELECT 1` via better-sqlite3 inside the route handler. Wrap in try/catch → `dbHealthy: false` on any error.
- **Latency target:** <50ms response time (enforced by having no filesystem/network calls beyond the DB probe).

### State persistence layout
- **Directory:** `.update-state/` at project root. Gitignored. Created on app boot if missing.
- **File:** `.update-state/state.json` holds cross-process update state. Format:
  ```json
  {
    "currentSha": "abc1234...",
    "rollbackSha": null,
    "lastCheckAt": null,
    "lastCheckEtag": null,
    "lastCheckResult": null,
    "updateStatus": "idle",
    "rollbackHappened": false,
    "rollbackReason": null
  }
  ```
- **Atomic writes:** Write to `state.json.tmp`, then `rename()` to `state.json`. No partial-write risk if the process dies mid-write.
- **Access:** `src/modules/self-update/update-state-store.ts` exposes `UpdateStateStore` class with `read()`, `write(patch)`, `init()` methods. Readable by both Node and the bash updater (JSON is universal).

### Drizzle schema: update_runs
```ts
// src/db/schema.ts addition
export const updateRuns = sqliteTable("update_runs", {
  id: integer("id").primaryKey({ autoIncrement: true }),
  startAt: integer("start_at", { mode: "timestamp_ms" }).notNull(),
  endAt: integer("end_at", { mode: "timestamp_ms" }),
  fromSha: text("from_sha").notNull(),
  toSha: text("to_sha"),
  status: text("status", { enum: ["running", "success", "failed", "rolled_back"] }).notNull(),
  stage: text("stage"),           // which stage was running when status changed
  errorMessage: text("error_message"),
  rollbackStage: text("rollback_stage"),  // "stage1" | "stage2" | null
});
```
- **Migration:** Use `drizzle-kit generate` then commit the new migration file under `drizzle/`. Do NOT hand-write.
- **Rationale for SQLite (not just state.json):** state.json is for current state the shell script can read; update_runs is for audit history the UI can query.

### UI scope for Phase 7
Minimal. Add a `<VersionBadge />` React client component that:
- Imports `CURRENT_SHA_SHORT`, `CURRENT_SHA`, `BUILD_TIME` from `@/lib/version`
- Renders short SHA + build date (e.g. `v abc1234 · 2026-04-10`)
- On hover shows full SHA via native `title` attribute
- On click: copies full SHA to clipboard and shows a transient "Kopiert ✓" state

Placement: top of the existing Settings page (wherever the user lands via the nav "Settings" entry). Subtle, not a new page.

**Explicitly NOT in this phase:**
- Update-available badge/banner (Phase 8)
- "Check now" button (Phase 8)
- Install button or update flow UI (Phase 10)

### Module layout
```
src/
├── lib/
│   └── version.ts                  (generated, git-ignored)
├── modules/
│   └── self-update/
│       ├── update-state-store.ts   (JSON file manager, atomic writes)
│       └── types.ts                (shared UpdateState, UpdateRun types)
├── db/
│   └── schema.ts                   (+ updateRuns table)
├── app/
│   ├── api/
│   │   └── version/route.ts        (GET version)
│   └── settings/
│       └── version-badge.tsx       (client component, tiny)
drizzle/
└── <timestamp>_update_runs.sql     (generated migration)
scripts/
└── build/
    └── generate-version.mjs        (prebuild script)
```

### Boot integration
- `server.ts main()` calls `UpdateStateStore.init()` early (before HttpPollingService) so every subsequent boot has a valid state.json.
- Init logic: if state.json exists → read; if missing → write the default record with `currentSha: CURRENT_SHA`. Never overwrite an existing file on boot.

### Claude's Discretion
- Exact file names within the module (can rename if cleaner).
- Whether UpdateStateStore uses a synchronous constructor (`new UpdateStateStore()`) or an async factory (`await UpdateStateStore.create()`) — synchronous preferred to match the rest of the codebase's pattern.
- Clipboard API choice: prefer `navigator.clipboard.writeText` with a fallback to the legacy `execCommand('copy')` path only if the settings page supports it.
- Test strategy for the prebuild script (probably integration test that runs `pnpm gen:version` in a tmp dir).

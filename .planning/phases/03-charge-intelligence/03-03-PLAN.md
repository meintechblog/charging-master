  <action>
    All routes use `export const runtime = 'nodejs'`.

    **POST /api/charging/learn/start (start/route.ts):**
    Accept JSON: `{ plugId: string, profileId: number }`.
    Validate: plugId exists in plugs table, profileId exists in deviceProfiles table.
    Check: No active learning session exists for this plug (query chargeSessions where plugId matches and state='learning').
    Create a new charge_session row: state='learning', plugId, profileId, startedAt=Date.now(), createdAt=Date.now().
    Call `globalThis.__chargeMonitor.startLearning(plugId, sessionId)` to put the plug's state machine into LEARNING state.
    Return 201 with `{ sessionId, plugId, profileId, state: 'learning' }`.
    Per D-28: Learning runs server-side -- this just triggers it.

    **POST /api/charging/learn/stop (stop/route.ts):**
    Accept JSON: `{ plugId: string, action: 'save' | 'discard' }`.
    Find active learning session for plugId (state='learning' or 'learn_complete').
    Call `globalThis.__chargeMonitor.stopLearning(plugId)`.

    If action === 'discard':
      Update session state='aborted', stoppedAt=Date.now(), stopReason='manual'.
      Delete session readings for this session.
      Return 200 with `{ ok: true, action: 'discarded' }`.

    If action === 'save':
      1. Fetch all sessionReadings for this session, ordered by offsetMs.
      2. Downsample to 1 reading per second: Group by Math.floor(offsetMs / 1000), take the last reading in each second bucket.
      3. Compute cumulative Wh: For each point, cumulativeWh += (apower * intervalSeconds / 3600). intervalSeconds = difference in offsetSeconds from previous point.
      4. Compute metadata: startPower = first point apower, peakPower = max apower, totalEnergyWh = last point cumulativeWh, durationSeconds = last offsetSeconds, pointCount = number of downsampled points.
      5. Delete any existing referenceCurve for this profileId (re-learn overwrites per D-32).
      6. Insert into referenceCurves with computed metadata.
      7. Bulk insert into referenceCurvePoints (downsampled points with offsetSeconds, apower, voltage, current, cumulativeWh).
      8. Compute SOC boundaries via `computeSocBoundaries(points)` and insert into socBoundaries table.
      9. Update charge_session: state='complete', stoppedAt=Date.now(), stopReason='learn_complete', energyWh=totalEnergyWh.
      Return 200 with `{ ok: true, action: 'saved', curveId, pointCount, totalEnergyWh, durationSeconds }`.

    Per D-27: The learn_complete auto-detection happens in ChargeMonitor. This endpoint is called when user confirms save/discard in the UI dialog.

    **GET /api/charging/learn/status (status/route.ts):**
    Query chargeSessions where state IN ('learning', 'learn_complete'), ordered by startedAt DESC.
    For each active session, include: sessionId, plugId, profileId, state, startedAt, duration (Date.now() - startedAt), readingCount (count sessionReadings), latestPower (last reading's apower), cumulativeWh (sum from readings).
    Return array (typically 0 or 1 items, but supports multiple plugs learning simultaneously).
    Per D-28: This is how the UI checks recording status after browser reopen.
  </action>

  <action>
    All routes use `export const runtime = 'nodejs'`.

    **GET /api/charging/sessions (sessions/route.ts):**
    Query chargeSessions where state NOT IN ('complete', 'aborted', 'error', 'idle'). These are active/ongoing sessions.
    Left-join deviceProfiles to include profileName.
    Left-join plugs to include plug name.
    Return array of session objects with: id, plugId, plugName, profileId, profileName, state, detectionConfidence, targetSoc, estimatedSoc, energyWh, startedAt, durationMs (Date.now() - startedAt).

    **GET /api/charging/sessions/[id] (sessions/[id]/route.ts):**
    Fetch single session with profile and plug info.
    Include recent sessionReadings (last 60 readings for mini-chart).
    Return 404 if not found.

    **PUT /api/charging/sessions/[id] (sessions/[id]/route.ts):**
    Accept JSON: `{ profileId?: number, targetSoc?: number }`.
    Per CHRG-02: User can manually override the detected profile. If profileId provided, update session.profileId and load new profile's reference curve for matching.
    Per D-34: User can adjust target SOC. If targetSoc provided, validate 10-100 in steps of 10, update session.targetSoc.
    Call `globalThis.__chargeMonitor.overrideSession(sessionId, { profileId, targetSoc })` so the running state machine uses the new values. The overrideSession method (implemented in Plan 02) updates the state machine's targetSoc and reloads profile curve data from DB if profileId changed.
    Return updated session.

    **POST /api/charging/sessions/[id]/abort (sessions/[id]/abort/route.ts):**
    Per D-34: User can abort charging session entirely.
    Find session, verify it's in an active state (detecting, matched, charging, countdown).
    Call `globalThis.__chargeMonitor.abortSession(plugId)`.
    Update session: state='aborted', stoppedAt=Date.now(), stopReason='manual'.
    Return 200 with `{ ok: true }`.
    Note: This does NOT turn off the relay. The relay stays as-is. User uses existing relay toggle if they want to cut power.
  </action>

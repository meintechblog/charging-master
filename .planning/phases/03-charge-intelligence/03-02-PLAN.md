    **1. Extend EventBus (src/modules/events/event-bus.ts):**

    **2. Create curve-matcher.ts (src/modules/charging/curve-matcher.ts):**
    Implement `matchCurve` function as specified in RESEARCH.md:
    - Input: queryReadings (number[]), profiles (array with id, name, curvePoints, curve metadata)
    - Phase 1: Quick-reject by initial power range (25% tolerance on first 10 readings average vs profile startPower)
    - Phase 2: subsequenceDtw on remaining candidates
    - Confidence calculation: `Math.max(0, 1 - (distance / (avgQueryPower || 1)))`
    - Return MatchResult or null if best confidence < 0.70 threshold
    - Define ProfileWithCurve interface locally: `{ id: number; name: string; curve: { startPower: number; durationSeconds: number; totalEnergyWh: number }; curvePoints: Array<{ offsetSeconds: number; apower: number; cumulativeWh: number }> }`

    **3. Create charge-monitor.ts (src/modules/charging/charge-monitor.ts):**
    ChargeMonitor class (server-side singleton):
    - Constructor takes: eventBus, mqttService, db
    - `start()`: Listen to `power:*` on eventBus. On each reading, route to per-plug state machine.
    - `stop()`: Remove listeners, clear timers.
    - Internal `machines: Map<string, ChargeStateMachine>` -- lazy-created per plug.
    - Internal `detectionBuffers: Map<string, number[]>` -- accumulate readings during DETECTING state.
    - Internal `sessionWh: Map<string, number>` -- cumulative Wh per active session.
    - Internal `lastRelayOff: Map<string, number>` -- for hysteresis tracking.

    On power reading:
    - Get or create state machine for plugId
    - Feed reading to state machine
    - Handle transitions via onTransition callback:
      - IDLE -> DETECTING: Create charge_session row with state='detecting', clear detection buffer, start accumulating readings
      - DETECTING (after 60-120 readings accumulated): Run matchCurve against all profiles from DB. If match found, call machine.setMatch(). If no match after 120s, emit 'unknown device' event.
      - MATCHED: Update session with profileId, confidence, dtwScore. Emit charge:* event with state='matched'.
      - MATCHED -> CHARGING: Update session state. Start tracking cumulative Wh. Emit charge:* event.
      - CHARGING: On each reading, update sessionWh, compute estimatedSoc via estimateSoc(). Update session row. Emit charge:* event with updated SOC. Check if SOC >= targetSoc - 5 for countdown transition.
      - COUNTDOWN: Continue SOC tracking. Emit charge:* with state='countdown'. When SOC >= targetSoc, transition to STOPPING.
      - STOPPING: Call switchRelayOff(). If success, transition to COMPLETE and update session. If failure, transition to ERROR.
      - LEARNING -> LEARN_COMPLETE: Per PROF-04, auto-detect charge-complete when power drops to ~0W. (Processing of learn data is handled by learn API routes in Plan 03.)

    Per D-28 (CRITICAL): On startup, check DB for sessions with state='learning' or 'detecting' or 'charging' etc. Resume those sessions by recreating state machines in the correct state.

    Per Pitfall 2 (sparse Shelly updates): During LEARNING and CHARGING states, use a setInterval to request status via MQTT RPC every 5 seconds: publish `{topicPrefix}/rpc` with payload `{"id":1,"src":"charging-master","method":"Switch.GetStatus","params":{"id":0}}`. Add a `requestStatus(topicPrefix: string)` method to MqttService for this.

    **4. Add requestStatus to MqttService (src/modules/mqtt/mqtt-service.ts):**

    requestStatus(topicPrefix: string): void {
      if (!this.client) return;
      this.client.publish(`${topicPrefix}/rpc`, JSON.stringify({
        id: Date.now(), src: 'charging-master',
        method: 'Switch.GetStatus', params: { id: 0 }
      }));
    }

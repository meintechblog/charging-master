<objective>
Ship FPD-01 (Stale-Power Watchdog). When `apower < charging.stalePowerThresholdW` (default 1.0 W) for >= `charging.stalePowerWindowSec / 5s` consecutive readings (default 300/5 = 60 readings) during `state ∈ {charging, countdown}`, the state machine fires an abort transition, ChargeMonitor switches the relay off, writes the session row with `stopReason='stale_power'`, and fires a Pushover anomaly (with the band's ASCII bar). The watchdog also emits a 'warning' SSE field starting at 20% of the configured window so FPD-05's UI banner can pre-display.

This is reading-based (counter increments only on actual sub-threshold readings), NOT wall-clock — a Shelly polling gap pauses the counter naturally (see RESEARCH Pitfall 1, 2 and CONTEXT design-decision 1). The abort path does NOT go through `handleStopping` (Pattern 2 in PATTERNS.md); it mirrors the existing `'learn_complete'` / `abortSession` direct-write path. No new ChargeState union member is added — destination state is `'aborted'`, reason is conveyed via `onTransition` callback `data` (resolved Open Question OQ-1).

Foundation plan. 12-02, 12-03, 12-04 depend on the new watchdog fields on ChargeStateEvent and the readStalePower* config helpers.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Shelly→backend (HTTP polling) | Untrusted apower readings cross here; malformed readings could be replayed. |
| backend→Pushover (HTTP egress) | Anomaly body contains a profile name (user-provided string); CRLF injection / Pushover-format-leak risk. |
| state-machine→ChargeMonitor (`onTransition data`) | New `data` payload becomes part of the abort write — must be type-narrowed before string coercion. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-01 | Tampering | ChargeStateMachine.transition data | mitigate | Type-narrow `data` in handleTransition via `data && typeof data === 'object' && 'reason' in data`. Only accept hard-coded reason strings (`'stale_power'`). Reject everything else (fall through to no-op). |
| T-12-02 | Information disclosure | Pushover body | accept | profileName is already injected into existing fireAnomalyNotification; same trust boundary, no new exposure. |
| T-12-03 | Denial of service | Pushover rate-limit (10k/month app key) | accept | At one stale-power abort per session, well under the 10k/month ceiling even with a runaway test scenario. |
| T-12-04 | Repudiation | stop_reason='stale_power' provenance | mitigate | Logged to chargeSessions table + Pushover history; auditable. |
</threat_model>

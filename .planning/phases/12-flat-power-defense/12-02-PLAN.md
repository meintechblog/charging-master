<objective>
Ship FPD-02 (Adaptive Matcher Refresh) and FPD-03 (Band-Confidence-Aware Stop Fallback) together. They share the `updateSocTracking` code path inside ChargeMonitor and the same `bandConfidence` field, so combining keeps the refresh + fallback decision tree in one place.

**FPD-02 mechanics:** During `state === 'charging'`, ChargeMonitor accumulates apower readings into a new in-memory `chargingBuffers: Map<plugId, { apower: number[]; profile: ProfileWithCurve }>` (resolved Open Question OQ-2: in-memory, not DB queried per-reading — see RESEARCH Pitfall 14 + Anti-pattern A; profile cached on first refresh to avoid re-querying DB every 60 readings). Every `matcherRefreshReadings` readings (default 60), it re-runs `findBestCandidate` against the buffer. The returned MatchResult is gated by monotonic narrowing (resolved Open Question PATTERNS-Q3: `Math.max(priorSocMin, candidateSocMin)` / `Math.min(priorSocMax, candidateSocMax)`). The existing in-place clamp at `charge-monitor.ts:715-731` is REUSED via inline math in refreshMatch — DO NOT extract a helper (see Task 2 action for rationale).

**FPD-03 mechanics:** Per RESEARCH §FPD-03 Q1, the "v1.2 energy-based fallback" formula `cumulativeEnergyWh >= (targetSoc - estimatedStartSoc)/100 * totalEnergyWh` is mathematically identical to `estimatedSoc >= targetSoc` (proved by inverting the estimateSoc formula). So the fallback predicate is trivial. When `bandConfidence < lowConfidenceThreshold`, ChargeMonitor calls `shouldStopEnergyFallback(estimatedSoc, targetSoc)` BEFORE the state machine's `feedReading` runs (so the band-mode shouldStop never gets a chance to fire on an untrusted band). If true, it dispatches the same stop path as a band-mode stop (transition `'charging'` → `'stopping'`) but tags the event with `stopMode='energy_fallback'`.

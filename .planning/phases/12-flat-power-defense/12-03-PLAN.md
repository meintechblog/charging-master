<objective>
Ship FPD-04 (Session Max-Duration Watchdog). Different mechanism than FPD-01: this watchdog is **wall-clock based** (RESEARCH Pitfall 10). Rationale: a session that survived 24h is by definition pathological — we want it killed regardless of whether readings have been arriving. FPD-01 (reading-based) covers "lots of zero readings". FPD-02 (in-memory matcher refresh) covers "matcher confidence isn't growing". FPD-04 catches the residual case "system has been spinning on this session for a day, kill it" — which is why a polling gap should NOT pause this counter.

`sessionStartedAt` is already in `ChargeMonitor.sessionStartedAt: Map<plugId, number>` (line 70) and is already restored on resume (`resumeActiveSessions` line 1245). So FPD-04 reads from existing state — no schema change, no resume-path change.

The abort path mirrors 12-01's stale-power path: transition to `'aborted'` via `onTransition` callback `data={reason:'timeout'}`. ChargeMonitor's handleTransition('aborted') case (built in 12-01) is extended to route by reason: `'stale_power'` → fireStalePowerNotification; `'timeout'` → new fireTimeoutNotification.

Wave 2. Independent of 12-02 — pure wall-clock + Pushover. Could theoretically have been merged into 12-02, but kept separate so 12-02 stays focused on the matcher+fallback bundle.
</objective>

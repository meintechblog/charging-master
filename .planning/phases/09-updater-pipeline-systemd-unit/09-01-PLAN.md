**Endpoint shape:**
- Method: `POST`
- Path: `/api/internal/prepare-for-shutdown`
- File: `src/app/api/internal/prepare-for-shutdown/route.ts`
- Runtime: `nodejs`
- Dynamic: `force-dynamic` (must not be cached)

**Host guard (MANDATORY — security):**
- Read `request.headers.get('host')` (lowercase per HTTP spec — Next.js gives it lowercased)
- Strip the optional `:port` suffix via `host.split(':')[0]`
- Accept only: `'127.0.0.1'`, `'localhost'`, `'::1'`
- Anything else → HTTP 403 with `{ error: 'forbidden' }`, do NOT proceed to drain or checkpoint
- DO NOT trust `x-forwarded-for`. The production unit binds to `0.0.0.0:80` with no reverse proxy, so the Host header is the only gate.
- The updater script hits `http://127.0.0.1:80/...` — which arrives at the handler with `host: "127.0.0.1"` or `host: "127.0.0.1:80"`.

**Drain sequence (in order):**
1. Host guard
2. Grab `globalThis.__httpPollingService` — if undefined (dev HMR edge case), return 500 with `{ error: 'polling_service_unavailable' }`. Do NOT crash.
3. Call `await httpPollingService.stopPolling()` (the new no-arg overload). Capture the returned count.
4. Run `sqlite.pragma('wal_checkpoint(TRUNCATE)')` synchronously. The `.pragma()` call returns an array of rows; `row[0]` has shape `{ busy, log, checkpointed }` — capture `checkpointed` as `drainedPages`.
5. Return 200 with `{ status: 'drained', at: Date.now(), drainedPages, pollersStopped }`.

**HttpPollingService.stopPolling() no-arg overload (MANDATORY):**
- Signature: `async stopPolling(): Promise<number>` — returns count of pollers that were active when the call started.
- Behavior:
  1. Snapshot `this.pollers.size` as `count`
  2. Clear every interval via the existing `stopAll()` logic (or inline the same loop — do not call `stopAll()` recursively if the overload is on the same method name; see Task 1)
  3. Wait a short settle window (`await new Promise(r => setTimeout(r, 100))`) so any fetch that fired just before the interval was cleared has a chance to resolve and hit `eventBus.emitPowerReading()` + the (potentially racy) DB update
  4. Return `count`
- Total wall clock: ~100ms. The 5s hard timeout mentioned in 09-CONTEXT.md is a generous upper bound — in practice settle is sub-second because the Shelly HTTP poll itself has `AbortSignal.timeout(3000)`, so the worst case is that a 3s-in-flight fetch is still pending; the settle window does NOT need to match the fetch timeout because better-sqlite3 writes are serialized by the DB engine and a post-checkpoint write is safe (it just ends up in a new WAL page). The P3 concern is about pending writes at the moment of `SIGKILL`, not at the moment of WAL checkpoint.

**Two overloads of stopPolling:**
TypeScript method overloads. Both must be callable:
```ts
stopPolling(plugId: string): void;  // existing — untouched logic
stopPolling(): Promise<number>;      // new — drains all
```
The implementation uses `function overloads` syntax with a union parameter, then branches at runtime.

**Response shapes:**
- 200 success: `{ status: 'drained', at: <epoch ms>, drainedPages: <number>, pollersStopped: <number> }`
- 403 non-localhost: `{ error: 'forbidden' }`
- 500 polling service missing: `{ error: 'polling_service_unavailable' }`
- 500 checkpoint error: `{ error: 'wal_checkpoint_failed', message: '<err.message>' }`
- All responses include `Cache-Control: no-store`

<pitfall_mitigation>
<!-- Phase 9 Plan 09-01 must explicitly mitigate P3 and P18 from .planning/research/PITFALLS.md. -->

- **P3 (better-sqlite3 / SQLite during the restart window):** The primary mitigation. `PRAGMA wal_checkpoint(TRUNCATE)` forces every committed WAL page into the main DB file and truncates the `-wal` file to zero. After this, the main DB file is self-contained — any subsequent SIGKILL on the node process cannot corrupt or silently drop data, because there's nothing in the WAL to lose. We also stop the HttpPollingService first, so no new writes land between the pre-checkpoint snapshot and the checkpoint itself. The small settle window (100ms) covers in-flight fetches.
- **P18 (silent success — HTTP 200 does not mean the app is really up):** This endpoint's 200 response is *part* of the structural guarantee that prevents P18 on the restart side. A 200 here means "polling stopped, WAL checkpointed, safe to stop" — three concrete invariants. If any one fails, we return a non-200 and the updater script's `curl -sf` refuses to proceed. No silent success possible.
- **Security boundary (CONTEXT §Security):** The Host-header guard is the only thing preventing a device on the same LAN from POSTing to this endpoint. An attacker on 192.168.1.x can reach port 80 but their Host header will be `192.168.1.x` or the machine's hostname — not `127.0.0.1` or `localhost`. The guard rejects them. We do NOT use any crypto (single-user local app) and do NOT use `x-forwarded-for` (not a reverse-proxied deployment).
</pitfall_mitigation>

```ts
  /**
   * Stop polling a specific plug (single-plug overload).
   */
  stopPolling(plugId: string): void;
  /**
   * Stop ALL active polling intervals and wait for a brief settle window so
   * any in-flight HTTP fetch has a chance to complete before the caller moves
   * on to e.g. a WAL checkpoint. Returns the number of pollers that were
   * active at the moment the call started.
   *
   * Added in Phase 9 (EXEC-04) for the /api/internal/prepare-for-shutdown
   * drain path. The existing stopAll() method remains for the synchronous
   * SIGTERM shutdown path in server.ts — do not merge them.
   */
  stopPolling(): Promise<number>;
  stopPolling(plugId?: string): void | Promise<number> {
    if (typeof plugId === 'string') {
      // Single-plug overload — existing behavior, unchanged.
      const timer = this.pollers.get(plugId);
      if (timer) {
        clearInterval(timer);
        this.pollers.delete(plugId);
      }
      return;
    }

    // No-arg overload — drain-all.
    const count = this.pollers.size;
    for (const timer of this.pollers.values()) {
      clearInterval(timer);
    }
    this.pollers.clear();
    // Brief settle window so any fetch() that fired just before the interval
    // was cleared has a chance to resolve and land its DB write BEFORE the
    // caller checkpoints the WAL. 100ms is empirically enough — the fetch
    // itself already has AbortSignal.timeout(3000) but we don't need to wait
    // for a stuck fetch, only for "normal" in-flight ones.
    return new Promise<number>((resolve) => {
      setTimeout(() => resolve(count), 100);
    });
  }
```

```ts
import { sqlite } from '@/db/client';

export const runtime = 'nodejs';
export const dynamic = 'force-dynamic';

// Hosts that are allowed to call this endpoint. Production uses 127.0.0.1:80,
// dev uses 127.0.0.1:3000 / localhost:3000. The :port suffix is stripped
// before comparison. IPv6 loopback is accepted for completeness even though
// the updater never hits it.
const ALLOWED_HOSTS = new Set(['127.0.0.1', 'localhost', '::1', '[::1]']);

// Wall-clock hard ceiling: the updater's curl has --max-time 10, so we have
// to respond before that. Drain is ~100ms + checkpoint is <1s on a healthy DB,
// so 10s is generous. We enforce a 5s soft ceiling on the drain itself so a
// stuck fetch cannot block us indefinitely.
const DRAIN_HARD_TIMEOUT_MS = 5_000;

type DrainOk = {
  status: 'drained';
  at: number;
  drainedPages: number;
  pollersStopped: number;
};

type DrainErr =
  | { error: 'forbidden' }
  | { error: 'polling_service_unavailable' }
  | { error: 'drain_timeout' }
  | { error: 'wal_checkpoint_failed'; message: string };

const NO_CACHE_HEADERS = {
  'Cache-Control': 'no-store, no-cache, must-revalidate',
} as const;

function isLocalhostHost(request: Request): boolean {
  const raw = request.headers.get('host');
  if (!raw) return false;
  // Strip :port suffix. Handle both `127.0.0.1:80` and `[::1]:80`.
  // Host header is already lowercased by Next.js Headers API.
  const host = raw.startsWith('[')
    ? raw.slice(0, raw.indexOf(']') + 1)
    : raw.split(':')[0];
  return ALLOWED_HOSTS.has(host);
}

/**
 * POST /api/internal/prepare-for-shutdown
 *
 * Called by scripts/update/run-update.sh right before `systemctl stop
 * charging-master` so the app can:
 *   1. Stop every HttpPollingService interval so no new writes hit SQLite.
 *   2. Wait a brief settle window for in-flight Shelly fetches.
 *   3. Run `PRAGMA wal_checkpoint(TRUNCATE)` to force every committed WAL page
 *      into the main .db file, eliminating the P3 (PITFALLS.md) corruption
 *      window during the systemd stop.
 *
 * Mitigates Pitfall 3 (SQLite WAL recovery races) and Pitfall 18 (silent
 * success — the updater must know the DB is quiescent before stopping).
 *
 * Security: Host-header guard only. Rejects anything that isn't 127.0.0.1 /
 * localhost / ::1. No auth beyond that — we rely on network segmentation
 * (LAN-only deployment, single-user) + the guard preventing a neighbor LAN
 * device from sending a bogus drain request.
 */
export async function POST(request: Request): Promise<Response> {
  if (!isLocalhostHost(request)) {
    const body: DrainErr = { error: 'forbidden' };
    return Response.json(body, { status: 403, headers: NO_CACHE_HEADERS });
  }

  const httpPollingService = globalThis.__httpPollingService;
  if (!httpPollingService) {
    // Can happen during dev HMR or a half-booted test run. The updater script
    // treats any non-200 as a drain failure and aborts the update.
    const body: DrainErr = { error: 'polling_service_unavailable' };
    return Response.json(body, { status: 500, headers: NO_CACHE_HEADERS });
  }

  // Drain pollers with a hard timeout so a runaway settle cannot block us.
  let pollersStopped: number;
  try {
    pollersStopped = await Promise.race<number>([
      httpPollingService.stopPolling(),
      new Promise<number>((_, reject) =>
        setTimeout(() => reject(new Error('drain_timeout')), DRAIN_HARD_TIMEOUT_MS),
      ),
    ]);
  } catch (err) {
    const body: DrainErr = { error: 'drain_timeout' };
    return Response.json(body, { status: 500, headers: NO_CACHE_HEADERS });
  }

  // Force WAL → main DB. better-sqlite3's pragma() returns an array of rows;
  // the first row has { busy, log, checkpointed }. We report `checkpointed`
  // as drainedPages.
  let drainedPages = 0;
  try {
    const rows = sqlite.pragma('wal_checkpoint(TRUNCATE)') as Array<{
      busy: number;
      log: number;
      checkpointed: number;
    }>;
    drainedPages = rows[0]?.checkpointed ?? 0;
  } catch (err) {
    const body: DrainErr = {
      error: 'wal_checkpoint_failed',
      message: err instanceof Error ? err.message : String(err),
    };
    return Response.json(body, { status: 500, headers: NO_CACHE_HEADERS });
  }

  const body: DrainOk = {
    status: 'drained',
    at: Date.now(),
    drainedPages,
    pollersStopped,
  };
  return Response.json(body, { status: 200, headers: NO_CACHE_HEADERS });
}
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LAN → port 80 (prod) / 3000 (dev) | Any device on the local network can reach the Next.js port. Only the `isLocalhostHost()` guard prevents them from calling `/api/internal/prepare-for-shutdown`. |
| Browser → route handler | A user's browser cannot send a cross-origin POST with an arbitrary Host header (CORS + browser Host restriction), but the endpoint explicitly returns 403 on non-localhost regardless. |
| HttpPollingService → better-sqlite3 | Pollers write power readings directly. After the drain call returns, no pollers are running, so no writes can hit the DB until the service is restarted. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-01 | Spoofing | `/api/internal/prepare-for-shutdown` (LAN attacker) | mitigate | Host-header guard rejects any Host other than `127.0.0.1` / `localhost` / `::1`. A LAN attacker CANNOT forge a `Host: 127.0.0.1` header via a browser (browsers set Host from the URL). They CAN via raw curl, but to do so they must already be on a device inside the LAN AND choose to bind localhost — at which point they are the user. Accepted residual risk. |
| T-09-02 | Tampering | HttpPollingService state (stopPolling mid-fetch) | mitigate | The new no-arg `stopPolling()` awaits a 100ms settle window after clearing intervals so in-flight fetches can resolve. The existing fetch has `AbortSignal.timeout(3000)` so even a stuck fetch self-terminates well before any SIGKILL. |
| T-09-03 | Repudiation | Drain endpoint called but DB not actually checkpointed | mitigate | The 200 response carries `drainedPages` from the actual pragma result. The updater script logs this to journalctl. On a rollback, the run row preserves the stage so we know whether drain ran. |
| T-09-04 | Information disclosure | Error messages leak stack traces or internal paths | mitigate | All 4xx/5xx responses use fixed error codes (`forbidden`, `polling_service_unavailable`, `drain_timeout`, `wal_checkpoint_failed`). Only `wal_checkpoint_failed` includes the raw `err.message` — better-sqlite3 messages are short and do not contain paths. Accepted. |
| T-09-05 | Denial of service | Repeated drain calls from localhost tank the server | accept | Single-user app on localhost; if the user DoSes themselves via a shell loop, that's self-harm. No rate limit added (would complicate the updater's retry logic with no real benefit). |
| T-09-06 | Elevation of privilege | Drain endpoint allows running arbitrary SQL | mitigate | The endpoint runs exactly one hard-coded pragma. No user input reaches sqlite. No query builder, no template. |
| T-09-07 | Spoofing | `x-forwarded-for` spoofing to bypass localhost guard | mitigate | The guard uses ONLY `request.headers.get('host')`. It never reads `x-forwarded-for`. This is explicitly documented in the code comment. |

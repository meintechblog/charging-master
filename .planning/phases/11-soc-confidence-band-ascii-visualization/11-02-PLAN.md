<objective>
Persist the SOC band end-to-end inside the runtime: a new Drizzle migration adds `soc_min`, `soc_max`, `band_confidence` columns to `charge_sessions`; `updateSocTracking` propagates the three band anchors forward in lock-step (no narrowing from Wh alone — only narrows on new matcher runs); `captureEventContext` snapshots the band fields before the relay-off await (fixes the 2640873 bug-class for the bar); resume-after-restart reads the columns back (NULL → degrade to zero-width); the override API path collapses the band to zero-width on user correction; and the state machine reads a new `charging.stopMode` config row to choose between conservative (`socMin >= target`) and aggressive (`socMax - socMin <= 5 AND socBest >= target`) stop logic. The aggressive ordering pitfall is locked in by a dedicated unit test. A new integration test PROVES the runtime stops within <30s of band collapse (CONTEXT DoD bullet 3, B2 closed). Plan 11-01 left the MatchResult band fields OPTIONAL; once every producer is wired in Task 3a, the fields are tightened to required.

Purpose: Without persistence + propagation + stop-mode wiring, the band is just a number on a transient MatchResult. This plan makes it the runtime source of truth and exposes the configurable behavior that prevents the iPad-style mis-stop. The integration test in Task 3b is the existence proof for the DoD <30s claim.

Output: A modified runtime (state machine + monitor + override route), a new `stop-mode` module, a Drizzle migration, comprehensive Vitest coverage of the propagation + decision + resume code paths, AND an integration test proving the <30s aggressive-stop guarantee.
</objective>

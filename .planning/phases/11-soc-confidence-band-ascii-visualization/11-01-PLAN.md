<objective>
Refactor the DTW core so it surfaces every offset's distance, then derive a SOC confidence band `{ socMin, socMax, socBest, bandConfidence }` inside `findBestCandidate`. Extend the `MatchResult` type ADDITIVELY with the band fields as OPTIONAL so no downstream consumer breaks (Plan 11-02 will tighten them to required after every producer is wired). Keep `estimatedStartSoc` as the alias for `socBest`. Commit a synthetic-iPad-shaped reference-curve fixture and a deterministic generator so a Vitest property test can prove the band collapses to <= 5 % once the live query reaches the taper region. CRITICAL: the property test ALSO runs an empirical-calibration sweep over `[0.05, 0.10, 0.15, 0.20, 0.30]` and pins `DEFAULT_BAND_THRESHOLD_PCT` to the smallest threshold that collapses the band to ≤ 5 % in the taper region — eliminating the guessed default flagged in RESEARCH.md Pitfall 4 / Open Q1. This is the math foundation that every other plan in Phase 11 builds on.

Purpose: Today's `subsequenceDtw` discards every offset except the global minimum. On a flat-power phase (root cause of the 2026-05-13 iPad mis-stop) that single minimum is structurally ambiguous and the app commits to a random start-SOC. By returning the full distances vector and thresholding "within X % of best", the matcher exposes the *set* of plausible starts as a band; later plans propagate, render, and act on that band. The empirical calibration in Task 2 ensures the default threshold isn't a guess.

Output: An additive refactor of `dtw.ts`, `curve-matcher.ts`, and `types.ts`; a new fixtures directory with a synthetic-iPad-shaped curve; comprehensive unit + property tests gated by Vitest; an empirically-pinned `DEFAULT_BAND_THRESHOLD_PCT`.
</objective>

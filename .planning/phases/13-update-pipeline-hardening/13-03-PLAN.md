`UpdateStateStore.write(patch)` (per Plan 13-01's locked invariants):
- Spread-merge preserves every field not in the patch.
- The DELETE patch is exactly `{ lastQuarantine: null }`. All other fields survive.

State store `read()` is a synchronous `JSON.parse` of state.json. If `state.lastQuarantine` is null/undefined, the DELETE has nothing to remove — return 200 with `removedPath: null` (idempotent no-op shape per CONTEXT.md §PIPE-03 admin-page acceptance: "after delete, page reloads, banner clears" — works whether anything was actually deleted).

Filesystem op: `import { rm } from 'node:fs/promises'` then `await rm(state.lastQuarantine.path, { recursive: true, force: true })`. Per `recursive: true, force: true`: removes the dir tree; ignores ENOENT (race-free if the operator already manually rm'd it). Per RESEARCH Pitfall 8 (anti-pattern): runtime MUST be `'nodejs'` (NOT `'edge'`) because node:fs/promises requires it.

Optional: count files removed BEFORE rm so the response can include `removedFileCount`. Use `readdir(path, { recursive: true, withFileTypes: true })` + filter `.isFile()`. Wrap in try/catch — file-count is informational, not load-bearing. Mirrors the `drainedPages` info field on prepare-for-shutdown's response.

Per CONTEXT.md §PIPE-03 + RESEARCH §PIPE-03 §approach §4: response shape on success is `Response.json({ ok: true, removedPath: <abs-path-or-null>, removedFileCount: <int-or-null> })`. On guard failure: `Response.json({ error: 'forbidden' }, { status: 403 })`. On rm or state write failure: `Response.json({ error: 'rm_failed' | 'state_write_failed', message }, { status: 500 })`.

Path safety: `state.lastQuarantine.path` is the ABSOLUTE path the bash updater wrote (Plan 13-01 Task 4 confirmed this). It must be inside `/opt/charging-master/.update-state/` per the bash logic. Defense-in-depth: assert the path string startsWith `process.cwd()` + `/.update-state/quarantine-` before rm. If not, return 500 `{error:'path_not_in_state_dir'}`. This catches the (hypothetical) corruption case where a malicious or buggy state.json points elsewhere — a real `rm -rf` on an arbitrary path would be catastrophic.

`src/db/client` is NOT imported here — this endpoint touches state.json + filesystem only, no DB. Per CONTEXT.md §Design Decision 6: PIPE-04 endpoint writes a `update_runs` audit row, PIPE-03 endpoint does NOT. The CONTEXT explicitly limits PIPE-04's side-effects; PIPE-03 is intentionally cheaper still.

JSDoc block above the handler explaining:
- Purpose: clear the most-recent preflight quarantine dir + reset state.lastQuarantine.
- Called by: the `quarantine-list.tsx` client component on the admin page (Plan 13-04) via `fetch('/api/admin/update-state/quarantine', { method: 'DELETE' })`.
- Host guard: `isAllowedBrowserHost` (LAN browser hostnames allowed, per RESEARCH Pitfall 12).
- Side effects: filesystem rm-rf of `state.lastQuarantine.path` + state.json patch `{ lastQuarantine: null }`. No DB write (per CONTEXT.md §Design Decision 6).
- Idempotent for the no-quarantine case (returns 200 with `removedPath: null`).

DELETE handler:
```ts
export async function DELETE(request: Request): Promise<Response> {
  if (!isAllowedBrowserHost(request)) {
    return Response.json({ error: 'forbidden' }, { status: 403, headers: NO_CACHE_HEADERS });
  }

  const store = new UpdateStateStore();
  let state;
  try {
    state = store.read();
  } catch (err) {
    return Response.json(
      { error: 'state_read_failed', message: err instanceof Error ? err.message : String(err) },
      { status: 500, headers: NO_CACHE_HEADERS },
    );
  }

  const quarantine = state.lastQuarantine;
  if (quarantine === undefined || quarantine === null) {
    // No quarantine on record — idempotent no-op. UI can still call this to "make sure".
    return Response.json(
      { ok: true, removedPath: null, removedFileCount: null },
      { status: 200, headers: NO_CACHE_HEADERS },
    );
  }

  const target = quarantine.path;

  // Defense-in-depth: the path must live inside the project's .update-state/quarantine-* tree.
  // Catches a hypothetical state.json corruption where lastQuarantine.path points elsewhere —
  // a real rm -rf on an arbitrary path would be catastrophic.
  if (!target.startsWith(QUARANTINE_PATH_PREFIX)) {
    console.warn(
      `[DELETE /api/admin/update-state/quarantine] refusing rm: path "${target}" not under ${QUARANTINE_PATH_PREFIX}`,
    );
    return Response.json(
      { error: 'path_not_in_state_dir', message: target },
      { status: 500, headers: NO_CACHE_HEADERS },
    );
  }

  // Best-effort file count BEFORE rm. Informational only — failure does NOT block the rm.
  let removedFileCount: number | null = null;
  try {
    const entries = await readdir(target, { recursive: true, withFileTypes: true });
    removedFileCount = entries.filter((e) => e.isFile()).length;
  } catch {
    // Directory may have been hand-removed by an operator. Continue.
  }

  try {
    await rm(target, { recursive: true, force: true });
  } catch (err) {
    return Response.json(
      { error: 'rm_failed', message: err instanceof Error ? err.message : String(err) },
      { status: 500, headers: NO_CACHE_HEADERS },
    );
  }

  try {
    store.write({ lastQuarantine: null });
  } catch (err) {
    // The directory is already gone — partial failure surface. Surface 500 so the UI
    // does NOT show "success" while state.json still claims a quarantine exists.
    return Response.json(
      { error: 'state_write_failed', message: err instanceof Error ? err.message : String(err) },
      { status: 500, headers: NO_CACHE_HEADERS },
    );
  }

  return Response.json(
    { ok: true, removedPath: target, removedFileCount },
    { status: 200, headers: NO_CACHE_HEADERS },
  );
}
```

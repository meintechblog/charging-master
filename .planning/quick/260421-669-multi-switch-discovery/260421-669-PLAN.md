# Quick Task 260421-669 — Multi-switch Discovery (identify-only)

## Goal

`Shelly.GetStatus` on a multi-channel device (e.g. the user's 192.168.3.53)
exposes `switch:0`, `switch:1`, ... but our scanner only ever looks at
`switch:0`. Users can't see channel 1 in Discovery, which makes identify-
by-toggle impossible for multi-channel devices.

Enumerate every `switch:N` component the Shelly exposes and surface each
as its own row in the Geräte-Erkennung list. Each row gets its own
identify-toggle that flips that specific channel's relay. User can
toggle-test every switch to figure out which physical outlet is which.

**Registration (the Hinzufügen button) stays blocked for `channel > 0`**
until Phase 999.1 lands — the DB schema + polling + charge logic still
key off a single device/channel, and half-registering would silently
drop readings. The button is disabled for channels > 0 with a tooltip
pointing at 999.1.

## Scope

- `src/modules/shelly/discovery-scanner.ts` — probeDevice returns
  `ScanResult[]` (one per switch), not `ScanResult | null`.
- `src/modules/shelly/relay-http.ts` — accept optional `channelId`
  parameter.
- `src/app/api/devices/relay-by-ip/route.ts` — forward optional
  `channel` from the body.
- `src/components/devices/ip-relay-toggle.tsx` — accept optional
  `channel` prop, forward in POST body.
- `src/components/devices/discovery-list.tsx` — render one row per
  channel, unique key `${deviceId}:${channel}`, disable Hinzufügen
  for channel > 0 with tooltip.

No DB change. No schema migration. No changes to the sync
`/api/devices/discover` endpoint or to any registered-plug code.

## Tasks

### T1 — Scanner enumerates all switch:N

- **File:** `src/modules/shelly/discovery-scanner.ts`
- **Action:**
  - Change `probeDevice` return type to `Promise<ScanResult[] | null>`.
    `null` = not a Shelly / unreachable; `[]` = Shelly but no switch
    components (edge case); otherwise one entry per switch.
  - After `Shelly.GetDeviceInfo` succeeds, fetch `Shelly.GetStatus`
    and `Shelly.GetConfig` in parallel. From the status response,
    collect every key matching `^switch:(\d+)$` and extract
    `apower` / `output` per id. From the config response, extract
    the trimmed `name` per id.
  - Build one `ScanResult` per switch:
    `{ip, deviceId: id, model, gen, channel: N, channelName,
     apower, output}`. Sort by `channel` ascending.
  - If `Shelly.GetStatus` fails, fall back to the old single
    `Switch.GetStatus?id=0` path returning `[{channel: 0, ...}]` so
    older-firmware devices don't regress.
- **Also:** `ScanResult` gains `channel: number` field.
- **scanSubnet:** after `probeDevice` returns a non-null array,
  iterate each and push / `onDevice(r)` individually.

### T2 — Relay-by-ip + IpRelayToggle accept channel

- **Files:** `src/modules/shelly/relay-http.ts`,
  `src/app/api/devices/relay-by-ip/route.ts`,
  `src/components/devices/ip-relay-toggle.tsx`
- **Action:**
  - `switchRelayOnHttp(ip, channel = 0)` and
    `switchRelayOffHttp(ip, channel = 0)` build `?id=${channel}`.
    (Explicitly constrain channel to a non-negative integer before
    interpolation to keep the URL safe.)
  - `/api/devices/relay-by-ip` accepts optional `channel: number`
    in the body. Validate `Number.isInteger(channel) && channel >= 0
    && channel <= 7` (arbitrary reasonable upper bound). Forward to
    the relay helpers.
  - `IpRelayToggle` accepts optional `channel?: number` prop,
    includes it in the POST body when set.

### T3 — DiscoveryList: one row per switch + gated Hinzufügen

- **File:** `src/components/devices/discovery-list.tsx`
- **Action:**
  - State key for devices and relay-state becomes `${deviceId}:${channel}`.
  - Unique rendering per channel; primary label: channel name if
    present, otherwise `${device.deviceId} (Kanal ${channel})`.
  - Row's `IpRelayToggle` gets `channel={device.channel}`.
  - Hinzufügen button rendered as before **only when
    `device.channel === 0`**. For `channel > 0`, render a disabled
    badge/button with title "Multi-Channel-Support folgt in Phase
    999.1 — Kanal 0 kann bereits hinzugefügt werden."
  - `registeredIds.includes(deviceId)` still filters channel-0 rows
    out post-registration; channel > 0 rows of the same device keep
    showing (they're not registered anyway).

  verify `updateAvailable: false`.

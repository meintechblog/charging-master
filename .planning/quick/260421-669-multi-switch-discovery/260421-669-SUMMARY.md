# Quick Task 260421-669 — Summary

## Scope drift note

This quick task started as "identify-only": surface every `switch:N` of
a multi-channel Shelly in the Discovery list so the user can toggle
each to figure out which physical outlet is which. Registration of
non-zero channels was explicitly deferred to backlog phase 999.1.

When the user tested it on a real multi-channel device (192.168.3.53,
"Beleuchtung Spielturm" on channel 1), the "999.1"-gated badge next
to the Hinzufügen button was rightly called out as a useless
half-feature. The task scope expanded to cover the minimal 999.1
refactor in the same session.

## What changed

### Discovery enumeration (ec20a40)
- `probeDevice()` now calls `Shelly.GetDeviceInfo`, `Shelly.GetStatus`
  and `Shelly.GetConfig` in parallel, enumerates every
  `^switch:(\d+)$` key in the status payload, and returns one
  `ScanResult` per channel. Each carries a `channel: number` +
  `channelName: string | null` from the per-channel `Switch.GetConfig.name`.
- Older firmware that doesn't expose the multi-component shape falls
  back to the single-id=0 path, returning `[{channel: 0, ...}]` so no
  single-channel device regresses.
- `scanSubnet()` flattens the per-IP array, `onDevice` fires once per
  channel, the SSE stream emits a `device` frame per channel.

### Multi-channel registration (3bc75c9)
- Schema: `plugs.channel INTEGER NOT NULL DEFAULT 0`. Drizzle schema
  updated; LXC patched manually via `ALTER TABLE plugs ADD COLUMN
  channel INTEGER NOT NULL DEFAULT 0;` because `drizzle-kit` isn't
  part of the deploy flow yet. All three pre-existing plugs land on
  `channel=0` via the default, no data migration needed.
- Composite id convention: channel 0 keeps the plain deviceId,
  channel > 0 uses `${deviceId}:${channel}`. `charge_sessions.plug_id`
  and `power_readings.plug_id` keep pointing at `plugs.id` as before —
  the composite is just a different opaque string.
- `switchRelayOnHttp` / `switchRelayOffHttp` now take
  `(ipAddress, channel = 0)`, with `Number.isInteger && >= 0` coercion
  before URL interpolation so nothing funky lands in the query string.
- `HttpPollingService.startPolling(plugId, ip, intervalMs, channel)`;
  poll URL is `Switch.GetStatus?id=${channel}`.
- `/api/devices/[id]/relay` and `/api/devices/relay-by-ip` both honor
  `plug.channel` (resp. request body `channel`, validated 0-7 int).
- `ChargeMonitor`'s `stopping` + `learn_complete` handlers now pass
  `channel: plug.channel ?? 0` to `switchRelayOff`. Same for
  `/api/charging/learn/{start,stop}`.
- `POST /api/devices` accepts `channel` in the body and stores it;
  `PATCH` restarts polling with the stored channel.
- UI: `DeviceManager.handleAddFromDiscovery` builds the composite id
  (`${deviceId}:${channel}` for channel > 0) and forwards the channel
  in the POST body. `DiscoveryList` drops the 999.1-gated badge; the
  Hinzufügen button is always active. The registered-vs-unregistered
  filter consults the composite key so a partially-registered
  multi-channel device still offers its remaining channels.

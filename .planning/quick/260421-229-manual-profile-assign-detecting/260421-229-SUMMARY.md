# Quick Task 260421-229 — Summary

## What changed

One file touched: `src/components/charging/charge-banner.tsx`.

The `detecting` render branch (previously just a spinner + "Gerät wird
erkannt..." label) now also shows a compact "Profil manuell zuweisen"
control. Clicking it reveals the existing profile chip list — the same one
used in the active-state branch (lines ~426-447) — wired to the existing
`handleOverrideProfile(profileId)` callback. Selecting a chip fires
`PUT /api/charging/sessions/:id` with `{profileId}`, which routes through
`ChargeMonitor.overrideSession()` → synthesizes a match → pushes the state
machine to `matched`. The banner transitions on the next SSE charge event.

No schema changes. No new API. `UnknownDeviceDialog` and the existing
post-exhaustion flow remain untouched.

## Implementation notes

- Reused `profiles` state and `showProfileDropdown` toggle — both were
  already fetched/maintained for the active-state branch, so the new UI
  adds zero state overhead.
- Guarded on `session.sessionId && profiles.length > 0` so the control
  only renders when both a session id and at least one profile exist
  (prevents dead-click with no API target).
- Visual weight kept deliberately low (`text-xs`, neutral palette, muted
  top border) so it doesn't compete with the "Gerät wird erkannt..."
  spinner — it's a fallback lane, not the primary UX.

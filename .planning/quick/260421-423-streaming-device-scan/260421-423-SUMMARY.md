# Quick Task 260421-423 ‚Äî Summary

## What changed

### Scanner

- `scanSubnet()` grew an `onDevice?: (device) => void` callback that
  fires the moment a probe resolves with a non-null result, before
  the current batch finishes. `onProgress` still fires per batch.
- `probeDevice()` now also calls `Switch.GetConfig?id=0` and exposes
  the trimmed `name` field as `ScanResult.channelName: string | null`
  (null when the user never set one or the call fails).

### New SSE endpoint

- `src/app/api/devices/discover/stream/route.ts` ‚Äî GET returns a
  `text/event-stream` (`Cache-Control: no-cache, no-transform`,
  `X-Accel-Buffering: no`). Forwards:
  - `event: device` with a full `ScanResult` JSON per probe hit.
  - `event: progress` with `{scanned, total}` per batch.
  - `event: done` with `{}` when `scanSubnet()` resolves.
  - `event: error` on exception.
  Aborts via `request.signal` flip a `closed` flag so late callbacks
  can't `enqueue` into a closed controller. The old sync
  `/api/devices/discover` endpoint is untouched.

### UI

- `discovery-list.tsx` rewritten around `EventSource`:
  - `handleScan` is a `useCallback` that closes any prior stream,
    resets state, opens a new SSE connection, and wires per-event
    handlers (device / progress / done / error).
  - Rows appear one-by-one as `device` frames arrive.
  - A 1.5 px blue progress bar with a caption
    `X / 254 IPs gepr√ºft ¬∑ N%` renders while `scanning === true`.
  - Channel name (when present) is the primary `text-sm` label;
    device id falls back to a secondary monospace line. Without a
    channel name the device id stays as the primary label.
  - `onAddDevice` now passes the channel name (or the device id as
    fallback) as the third `defaultName` argument.
  - EventSource is cleaned up on unmount via a ref.
- `device-manager.tsx`: `handleAddFromDiscovery(deviceId, ip,
  defaultName?)` uses the Shelly-defined name as the plug's initial
  `name` when provided, otherwise keeps the old `name = deviceId`
  behavior.

## Follow-ups

- The registered-list still shows `plug.name` from the DB, not the
  Shelly's live `Switch.GetConfig.name`. If the user renames the
  Shelly admin-side later, our DB copy stays stale until they
  manually rename in our UI too. Syncing that is a nice-to-have,
  not in this task's scope.
- The "Switch.GetConfig" call doubles the probe count per IP (254 ‚Üí
  508). With 20-way concurrency and a 1.5 s timeout the total
  scan time stays bounded (~25 s worst case) but is meaningfully
  longer. If this becomes a UX problem, fold the name into a second
  targeted request issued only for Shelly-positive IPs (short-circuit
  for non-Shelly IPs that already failed `Shelly.GetDeviceInfo`).

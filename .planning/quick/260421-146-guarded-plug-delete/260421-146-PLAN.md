# Quick Task 260421-146 — Guarded plug delete + richer device list

## Goal

Fix the broken delete on `/devices` page. Today the button silently fails
because `plugs.id` is FK-referenced by `charge_sessions.plug_id` and
`power_readings.plug_id` with no cascade, so SQLite rejects the delete.

New UX per user decision:

- Delete button **disabled + grayed** while the plug has an active charge
  session (`detecting` / `matched` / `charging` / `countdown` / `learning`).
- Otherwise enabled; click opens a **two-step confirm** inline ("Wirklich
  löschen?") — second click fires the DELETE. If the user changes their
  mind, the confirm resets after a few seconds or when another row's
  button is clicked.
- Server-side DELETE tears down all plug-scoped history atomically.

## Scope

- `src/app/api/devices/route.ts` — DELETE handler rewrites
- `src/app/devices/device-manager.tsx` — UI (active-session tracking + confirm flow)

No schema change. No new API. No changes to the POST/PATCH/GET handlers.

## Tasks

### T1 — Server: active-session guard + cascade delete

- **File:** `src/app/api/devices/route.ts`
- **Action:**
  - Before deleting, query `charge_sessions` for rows with this `plug_id`
    in states `['detecting','matched','charging','countdown','learning']`.
    If any, return `409` with `{error: 'active_session', sessionId, state}`.
  - Wrap deletion in `db.transaction(() => { ... })`:
    1. `DELETE FROM charge_sessions WHERE plug_id = ?` — `session_readings`
       and `session_events` cascade via their existing FK `{ onDelete: 'cascade' }`.
    2. `DELETE FROM power_readings WHERE plug_id = ?`.
    3. `DELETE FROM plugs WHERE id = ?`.
  - Then stop HTTP polling via `globalThis.__httpPollingService.stopPolling(id)`
    (already in current handler, just keep it).
- **Verify:** DELETE with active session returns 409; DELETE on idle plug
  returns 200 and all related rows in the three tables are gone.
- **Done:** Endpoint handles both cases and never leaves orphaned readings.

### T2 — UI: disabled state + confirm + active-session tracking

- **File:** `src/app/devices/device-manager.tsx`
- **Action:**
  - Add a `Set<string>` of plug IDs with active sessions to component state.
  - On mount, fetch `/api/charging/sessions` and seed the set with each
    returned `plugId`.
  - Subscribe via `useChargeStream('*', …)`: on each event, if state is
    one of the active ones, add `plugId` to the set; if state is
    `idle` / `complete` / `aborted` / `error`, remove it.
  - Add a `confirmingDelete: string | null` state. First click on a non-
    disabled Delete button sets `confirmingDelete = plug.id` and changes the
    label to "Wirklich löschen?"; second click within the window runs the
    actual DELETE. A `setTimeout` resets after 4s; clicking another plug's
    Delete also clears the pending confirm.
  - Disable the button (`disabled + opacity-40 + cursor-not-allowed + no
    hover color change`) when `activeSessions.has(plug.id)`. Add a title
    tooltip "Ladevorgang aktiv — erst beenden".
  - On non-ok DELETE response, read the JSON error and surface it inline
    below the row for that plug (auto-dismiss after 4s). 409 copy: "Ein
    Ladevorgang ist aktiv — bitte erst beenden."
- **Verify:** With Session 8 in `detecting` on Schuppen, the Delete button
  is ausgegraut. After aborting the session, button becomes active.
  Clicking asks for confirmation, second click deletes the plug.
- **Done:** Disabled / confirm / delete flow works end-to-end; errors
  surface visibly.
